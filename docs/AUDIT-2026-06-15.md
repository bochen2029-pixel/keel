## 1. Executive summary

**Health verdict: GREEN — coherent, with low-blast-radius doc-drift and bounded correctness gaps, none of which break a contract, a golden, or an invariant at runtime on the default path.**

The codebase is structurally faithful to its canon. The ten contracts are frozen and green; the layer-import rule holds in every crate (each imports only the layers below it); the golden freeze-gate is live and enforcing; and the strongest invariant — I5 externalized verification — is genuinely non-bypassable and well-tested on the main self-driving loop (fail-closed on unresolved golden refs, config-fault on critical steps with no oracle evidence). The audit found **zero critical and zero high findings.**

What it did find is a **coherent cluster around one seam — the manual `--tier` override path** — plus **stale prose in the `CLAUDE.md` Build-state block**, plus a handful of **bounded, mostly-not-default-on correctness/robustness gaps** in the daemon, memory consolidation loop, and middleware precision.

- **The `--tier` override is the load-bearing finding.** It skips the router (and therefore the I3 force-local *gate*) and the I5 verify loop entirely. On that path `--sovereign`, `--critical`, and `--golden-ref` are silently ignored — flags an operator can type believing they confer protection that the path does not deliver. The deterministic privacy mask still scrubs PII before egress on that path (so structured secrets do not leak), and the over-protocol surface (`keel-serve`) is unaffected — every HTTP request still goes through the full guarded loop. But the gap between a categorical documented guarantee ("sovereign/PHI/raw-perception may never leave the box") and the override's behavior is real. Three of the medium and two of the low findings trace to this one seam.
- **The doc-drift is real but self-disclaimed.** Three `CLAUDE.md` Build-state lines lag completed work (freeze-gate re-stamp, `svc::memory` shipped, `mw::metrics` shipped under a different shape). The block explicitly warns readers to "Trust STATE.md + git, never this block," which caps the blast radius — but per `CLAUDE.md` rule #1 the drift should still be reconciled.
- **The correctness gaps are latent, not live.** The daemon's shared per-task budget and the memory-maintenance self-ingest loop are both real, both benign under the default config (heartbeat is local/$0; `--consolidate-every` defaults to 0), and both worth fixing before the relevant feature is enabled by default.

Net: this is a clean, honest, lean codebase. The defects are concentrated, bounded, and several are already tracked as deferred issues. The single thing worth treating as more than bookkeeping is closing the `--tier` sovereignty/verification escape hatch.

#### M1 · [invariants] I3 sovereign GATE (force-local) is bypassable via the manual `--tier` override; the egress MASK is not

The I3 force-local **gate** lives only in the router: `crates/keel-services/src/router.rs:44` forces local for raw perception and `:48` forces local for `DataClass::Sovereign | Phi`. The canon corroborates this is by design — the gate is the router's job (`KEEL_ARCHITECTURE.md:95`, "Gate enforced in the router; mask + audit in the chain"; §17 table at `:359`, "The force-local *gate* is the router's (§5, §9), not this mask").

The `--tier` override **skips the router entirely.** `crates/keel/src/main.rs:98-110` calls `keel::assemble` (the single-tier path), which constructs **no** `Router` (`crates/keel/src/lib.rs:183-202`); the kernel's `run_on` is self-documented as "skipping route + verify" (`crates/keel-kernel/src/engine.rs:258-268`). So `keel --tier frontier --sovereign "..."` is **not** force-localed by the gate.

The egress **mask** still applies on that path: `lib.rs:198` sets `egress = (tcfg.adapter != "local_llama")`, and `build_chain` (`lib.rs:247-254`) wires `PrivacyMiddleware` with that flag, so rung-1/2 deterministic scrubbing still runs before egress. **Net:** on manual override the deterministic rung-1/2 mask protects egress of structured PII/secrets, but the stronger categorical "this data must never leave the box at all" gate is not enforced. The mask is currently thin: rung-1 operator markers are an empty list (`lib.rs:70` and `:199`, `Redactor::new(vec![])`) and rung-3 is deferred (ISSUE-2).

This is **not** a documented carve-out. Docs note that `--tier` skips the router (`lib.rs:11-12`, `engine.rs:258`) and document the gate-vs-mask split, but nowhere acknowledge that pinning a cloud tier defeats the sovereign gate. The override is framed as a cold-start-avoidance convenience, not a sovereignty escape hatch.

**Sharpening finding (same seam):** on the override path the `--sovereign` flag is effectively a **silent no-op.** `main.rs:98-110` builds only a `GenerateRequest` and never constructs a `Step` (no `data_class` is ever set or read), so the `sovereign` bool parsed at `main.rs:71` is unused on that branch. An operator could type `keel --tier frontier --sovereign ...` believing it is protected while the flag does nothing — worse than a simple skip, because it manufactures a false sense of protection.

**Severity medium (not high):** single-operator L1 tool; the bypass requires the operator to *deliberately* type a cloud `--tier`, and the deterministic rung-1/2 mask still backstops structured PII/secret egress.

**Recommendation:** Make the override path **refuse a non-local tier** when `--sovereign` (or sovereign `data_class`) is set, **or** feed the override through the router's hard-constraint check first, so the I3 gate cannot be defeated by pinning a cloud tier.

#### M2 · [this-run-correctness] Consolidate and cold-eyes maintenance turns self-ingest into the lossless Tape

`engine.run` unconditionally records every turn to its internal Tape at step 8a (`crates/keel-kernel/src/engine.rs:244-246`), and the wired engine memory **is** the Tape (`crates/keel/src/lib.rs:116`). `keel consolidate`, `keel cold-eyes`, and the daemon self-consolidation all run their maintenance prompt through `engine.run` (`crates/keel/src/main.rs` `do_consolidation` ~`:314-322`, `run_cold_eyes` ~`:357`, daemon ~`:237`). `FileMemory::record` (`crates/keel-services/src/memory.rs:311-338`) appends the full `Trace` to the lossless Tape with **no guard** on `step.source`/`step.kind`/`step.ty`; `recent_traces` (`memory.rs:206-212`) and `summarize` (`memory.rs:216-227`) read every line back with no filter.

Consequences, all verified:
1. A consolidation/cold-eyes turn re-lands on the lossless Tape, and `result.content` is the **model-authored narrative** — so the narrative contaminates the factual register, directly violating the A6.1 separation (`memory.rs:48-54`, `set_narrative` comment `:137`) and canon §22.8 "Confusing the registers" (`KEEL_ARCHITECTURE.md:494`), which mandate the factual register stay lossless/externalized and never hold a model-authored summary.
2. The consolidation prompt embeds the entire recent Tape verbatim (`memory.rs:152-178`), so the recorded maintenance turn balloons and the **next** consolidation re-ingests the prior consolidation turn via `recent_traces` — a compounding self-reference loop that also pollutes Ring-2 working context for subsequent normal user turns (`summarize` surfaces the giant maintenance prompt as a "user" line).

The markers to key off already exist but are never consulted by any filter: `source: Some("memory")` (`memory.rs:359`) and `ty: "memory:consolidate"`. This is **not** documented as deferred — `docs/proposals/perpetual-memory.md:49-51` prescribes "capture sanctity = append EVERY turn synchronously," which is what produces the loop; the doc never contemplates the consolidation turn re-ingesting itself.

**Severity medium (downgraded from high):** memory is the canon's flagged highest-risk seam, shipped as a deliberately-small first cut (`memory.rs:9-10`); the full consolidation wiring is itself opt-in (`--consolidate-every` defaults to 0; `consolidate`/`cold-eyes` are manual CLI). No critical-output correctness invariant is breached at runtime (verifier/oracle untouched); the original lossless turns remain intact and distinguishable on the Tape. The harm is memory-quality degradation plus a real-but-not-default-on register-confusion/bloat loop.

**Recommendation:** Exclude memory-maintenance turns from the Tape append in `engine.run` step 8a, or have `FileMemory.record` skip them (key off `source == "memory"` / the maintenance `ty`); at minimum filter them from `recent_traces`.

**Files:** `crates/keel-kernel/src/engine.rs` (244–246) · `crates/keel-services/src/memory.rs` (48–54, 137, 206–227, 311–363) · `crates/keel/src/main.rs` (190, 237, 314–322, 337, 357) · `crates/keel/src/lib.rs` (116) · `docs/proposals/perpetual-memory.md` (49–51) · `KEEL_ARCHITECTURE.md` (494).

#### M3 · [this-run-correctness] Perpetual daemon shares one per-task budget across an unbounded run

`run_daemon` builds `ctx` once via `new_context` (seeding `task_budget = Some(manifest.cost.budget_per_task)`, default 5.0) and **reuses that single `ctx` across every tick** (`crates/keel/src/main.rs:187`, `:226-259`), only ever reassigning `ctx.trace_id` — never resetting `ctx.cost` nor re-seeding `ctx.task_budget` per tick. `engine.run` accumulates cost every turn (`crates/keel-kernel/src/engine.rs:171`, `ctx.cost.add(...)`, purely additive — `crates/keel-contracts/src/types.rs:353`). The cost gate returns `BudgetExceeded` once `budget_remaining() < hard_stop_at` (default 1.0) (`crates/keel-middleware/src/cost.rs:29-35`), and on that error the daemon's `Err` arm logs and **breaks** (`main.rs:254-257`).

So a paid perpetual daemon climbs to ~$4 cumulative spend, then **every turn fails and the loop dies** — it does not throttle. Adversarial checks for an intentional/documented/deferred carve-out all came back negative: canon §8/§9 and the I4 line (`KEEL_ARCHITECTURE.md:96`) never address a daemon sharing one per-task budget across an unbounded run; the ROADMAP A2 daemon entry and the entire ISSUE register contain no mention of it. The daemon has only ever run the local, $0 path — the benign case.

**Severity medium:** latent/benign under the default config (heartbeat routes to the zero-cost local tier, so `cost.total` stays 0 and `remaining` never drops). It manifests only on a non-default paid-tier perpetual run (e.g. `--kind core-wire`, or escalations to cheap-API/frontier). The hard-stop *does* fire — the cap is over-eager, not absent. Note for the operator: the canon does not actually decide whether a daemon is one task (shared lifetime budget) or N tasks (per-tick budget); the code makes that choice implicitly, and the implicit choice is the failure-prone one.

**Recommendation:** Reset `ctx.cost` (or re-seed a per-turn budget) each tick, **or** give the daemon its own run-lifetime budget.

#### L1 · [invariants] I1 audit durability is best-effort — `FileAuditSink` swallows write failures to stderr

`FileAuditSink::emit` drops a serialize failure (`crates/keel-middleware/src/audit.rs:120-122`) and a write/flush failure (`audit.rs:124-128`) to `eprintln!` and returns `Ok` — the call proceeds even if the I1 ledger line was never persisted. The `AuditSink` trait `emit` returns `()` (`audit.rs:50`), so a sink structurally cannot signal a persistence failure to the caller. So I1 is observable-in-memory and unbypassable in the chain (the event is always constructed and emitted), but the **durable** record — which I2/resume lean on as the system of record (`KEEL_ARCHITECTURE.md:279`, `:335`) — can be silently lost on a disk error.

**Severity low:** intentional and documented, not an oversight. The code doc (`audit.rs:97-98`) explicitly labels this "Best-effort... a hash-chained durable sink swaps in later," and the canon anticipates the exact swap (`KEEL_ARCHITECTURE.md:358`). I1's core guarantee still holds; only durable persistence is best-effort, and only on a rare serialize/disk error.

**Recommendation:** Track the deferred hash-chained durable sink as the path to true I1 durability; at minimum, surface a counter or a one-time loud warning when an audit write fails, so a degraded ledger is not silent.

#### L2 · [invariants] I4 governed — ENFORCED as a pre-call remaining-budget floor; a single call can overshoot by one call

`CostMiddleware` blocks pre-call when `ctx.budget_remaining() < hard_stop_at` (`crates/keel-middleware/src/cost.rs:29-35`), proven by `blocks_at_the_floor_before_calling_the_tier` (`cost.rs:98-103`, terminal never reached). The engine folds `result.cost` **after** the chain (`crates/keel-kernel/src/engine.rs:171`). Because `Middleware::handle` sees only `&Context` (`chain.rs`), it cannot pre-charge the in-flight call's cost — the gate is a floor on *remaining* budget, so a single call costing more than the remaining headroom is only detected on the post-call fold. The run can exceed `budget_per_task` by **at most one call's cost.** Bounded today since best-of-N (`amplify`) ships OFF (`n` forced to 1 at `engine.rs:166`). The router carries a projected-cost pre-check (`router.rs:51-60`) but it only fires when `step.projected_cost` is `Some`, which the genome default path leaves `None` — the pre-charge seam exists but is dormant. The canon does not document this overshoot boundary as intentional, so it is a genuine enforcement-precision gap (no invariant is *bypassed* — I4 still hard-stops, one call coarser).

**Severity low:** bounded to a single call's cost; the 1.0 floor below a 5.0 budget designs in headroom; `amplify` is OFF so the dangerous multi-sample case is inert.

**Recommendation:** Acceptable for the current single-shot regime. Before `amplify` is enabled, add a per-call projected-cost pre-charge (populate `Step.projected_cost` on the request path — it is already threaded into the router at `router.rs:52`) so a multi-sample turn cannot overshoot, or tighten `hard_stop_at` relative to the most expensive single call.

#### L3 · [invariants] I5 externalized — the `--tier` / `run_on` override skips verification entirely (no oracle on that path)

`Engine::run_on` returns the `GenerateResult` with **no** `oracle.verify` call (`crates/keel-kernel/src/engine.rs:260-268`); the CLI `--tier` path runs `asm.chain.run` and reports without a verdict (`crates/keel/src/main.rs:109-110`). This is documented as deliberate (`lib.rs:12`, "skips the router + the I5 loop by design"). The fail-closed and critical-config-fault guards (`engine.rs:205-222`) do not run because `run_on` is a different entry point.

The load-bearing (and **not** fully documented) part: `main.rs` parses `--critical` (`:62`) and `--golden-ref` (`:63`) unconditionally, but the `tier_override` branch (`:98-110`) never references `critical` or `golden_refs`. So `keel --tier frontier --critical --golden-ref X "..."` silently **discards** both flags, runs with no oracle, and prints no verdict — no error, no warning. A caller who explicitly asserted `--critical` can believe the turn was verified when it was not. The docs say the override "skips the I5 loop by design" but never that "`--critical` is silently void here" — a genuine footgun against I5.

**Severity low:** the over-protocol surface is unaffected — `crates/keel/src/bin/keel-serve.rs` has no tier-pinning at all; every HTTP request goes through `engine.run` (`:144`) with full I5 guards (critical/golden_refs/grammar all wired). `--tier` is an L1 single-operator manual override; the self-driving default is correct. It is a UX/flag-handling footgun, not a corruption of the verified path itself.

**Recommendation:** Either **reject** `--critical` / `--golden-ref` on the `--tier` override path with a clear error (so a caller cannot believe a manual-override turn was verified), or route such turns through `Engine::run` with a fixed-tier router so I5 still applies.

#### L5 · [bugs] Inconsistent mutex-poison policy: driver/mic panic where memory/sink/store recover

`keel-services/src/driver.rs:59,63,75` (`UserTurnDriver`), `:106` (`HeartbeatDriver::tick`), `:158` (`WatchDriver::poll`) and `keel-adapters/src/mic.rs:45,68` all use `.lock().expect("...poisoned")`, which **panics** on a poisoned mutex (a fallible runtime path: any prior panic while holding the guard poisons it). The same project deliberately chose the poison-**tolerant** pattern elsewhere — `memory.rs:106,138,316`, `trace_sink.rs:63` use `.lock().unwrap_or_else(|p| p.into_inner())`, each with a justifying comment (e.g. `trace_sink.rs:62`, "poison-tolerant: a panicked holder must not wedge the corpus"). The panic sites carry no poison-policy comment. So a single panicked turn permanently wedges the driver loop / capture organ, whereas the Tape and corpus survive — an inconsistent failure policy for the same hazard class. No lock is held across an `.await` in any of these (guards are sync-scoped), so this is panic-on-poison only, not a deadlock. A repo-wide search confirms the policy is nowhere documented — an unremarked inconsistency, not a deferred decision.

**Severity low:** the panicking guards protect only trivial ops (`VecDeque` push/pop, `Instant` compare, token compare, `Vec::extend`) with no user code on the critical section, so poisoning requires a panic mid-trivial-op; blast radius is a panic, not silent corruption or deadlock; and the wedge-able daemon select-loop that consumes these drivers is itself a deliberate not-yet-built follow-on (`driver.rs:19-20`).

**Recommendation:** Unify on the poison-tolerant `unwrap_or_else(|p| p.into_inner())` (or a documented decision to fail-fast) across `driver.rs` and `mic.rs`, so a poisoned lock degrades the same way everywhere.

## 5. Honest caveats

- **This is an AI-agent audit, not the test gate.** It is a structured adversarial read of the source plus invariant tracing — it does not replace `cargo test`, `cargo clippy`, fuzzing, or runtime/integration testing. Every finding here was re-verified against the code, but "verified by reading" is weaker than "verified by executing." Treat the recommendations as hypotheses to confirm at the gate, not as facts already proven on the running binary.
- **Findings are biased toward what is legible in source.** Concurrency races, performance cliffs, and integration-time failures (e.g. real llama-server/whisper behavior on the `#[ignore]`d live paths) are under-sampled by a static read. The absence of a finding in those areas is not evidence of their absence.
- **Severity is calibrated to an L1 single-operator tool.** The medium/low ceiling reflects that the dangerous paths require the *operator* to deliberately invoke them (`--tier`, `--consolidate-every`, a paid-tier daemon) and that no defect breaks a contract, a golden, or an invariant on the default self-driving path. A different threat model (multi-tenant, hostile input) would re-weight several of these upward — but that is explicitly out of scope per the canon (L1, personal tool, no multi-tenancy).
- **Line numbers are a snapshot at HEAD `04a6acd`.** They will drift as the code changes; treat the file + symbol names as the durable anchor and the line numbers as a starting point.
- **No code was changed by this audit.** Every recommendation is a proposal for the operator/next session to action under the normal session protocol (layer-check → budget-check → golden-freeze verify → `cargo test` green → one commit). The doc-drift fixes (M4, M5, L4, L6) and the `--tier` hardening (M1, L3) are the highest-value, lowest-risk first moves.
