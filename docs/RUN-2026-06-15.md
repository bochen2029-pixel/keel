## The daemon-hang incident (ISSUE-8) and the kill-safe workaround

While verifying A2 *live* on 2026-06-15, a 2-tick daemon capture sat at 10m44s. Diagnosis **by artifact, not recall**: the daemon itself ran fine and exited fast (traces `-0`/`-2` on disk, no `keel.exe` left running). The hang was the **shell capture** — on Windows cold-start the detached llama-server inherits keel's stdout/stderr pipe handle, so a `… 2>&1 | Out-String` blocks on an EOF that never comes. A gate-green detach patch in `kernel::lifecycle::launch` (stdin → null + `DETACHED_PROCESS`) did **not** resolve the live hang (and the tool timeout didn't fire either), so per the operator it was **reverted** (keep the checkpoint honest), **ISSUE-8** was filed, and live-daemon verification was **pivoted off** (A2 is already proven by artifact).

The **workaround was later found and confirmed working** during the A6.2 live validation: run live-model commands via `Start-Process -RedirectStandardOutput <file> -PassThru`, then `$p.WaitForExit(ms)` → `$p.Kill($true)` — a **reliable self-kill timer** — using a **file redirect, NOT `| Out-String`** (the pipe was what hung on the detached server's inherited handle). With this pattern `keel consolidate`, `keel cold-eyes`, and the three-turn flywheel run all cold-started, ran, and **exited clean with no hang**.
